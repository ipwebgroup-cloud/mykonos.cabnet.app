Inquiry Queue

Use the inquiry list to read the live situation before opening a record. Ownership, urgency, linked follow-up, latest brief state, and next review timing should be visible before you open deeper details.

Queue reading order
Scan owner, urgency, and follow-up first
Start with who owns the inquiry, whether follow-up is overdue or due today, and whether the guest still needs first-touch or proposal shaping before loyalty work begins.
Linked continuity
Use packet and next-review cues before opening history
Linked queue rows should let operators read the latest packet posture and revisit timing at a glance. Open loyalty history only when the compact row cue shows something that needs deeper context.
Filter discipline
Use the repeat-guest filter to isolate handoff work
Separate linked, transfer-ready, draft-ready, and queue-only rows so continuity work does not widen the whole queue. This keeps proposal shaping and continuity stewardship from competing visually.

Repeat-guest follow-up

This area tracks repeat-guest follow-up, referral potential, and revisit timing without mixing early drafts with active guest follow-up work.

List posture
Use the list for latest outcome, latest packet, and next review
The list should stay compact and scan-friendly. Operators should see the source inquiry backlink, continuity posture, and next review cue before opening a record.
Create mode safety
Keep inquiry-backed drafts distinct from manual drafts
If a loyalty record came from a real inquiry, keep the source reference visible. If it is a manual draft, keep that distinction explicit so operators do not mistake it for a fully linked continuity lane.
Workspace tabs
Compare source posture before editing continuity fields
Overview, Workspace, and History should all let operators compare source inquiry posture against current continuity posture before they widen retention notes or review packet strategy.

Queue-to-loyalty bridge

Use continuity only when it helps operators route real post-inquiry value. Closed or maturing records can move into loyalty, but the queue should still show whether that handoff already happened.

Linked to loyalty
Repeat-guest follow-up already owns this guest lane
The source inquiry already has a loyalty record. Operators can open the linked record or jump straight to history if they only need the latest continuity result.
Transfer-ready
Queue work is mature enough for continuity handoff
Use this when the inquiry is closed or stable enough that remembered-return value should be preserved in a continuity record rather than living only in the queue timeline.
Draft-ready
A seeded draft exists but still needs operator review
This is useful when an operator wants create-mode context preserved from the inquiry while still keeping the continuity lane clearly marked as draft or watch-level.
Queue-only
Stay in the live inquiry lane for now
Not every inquiry needs continuity immediately. Leave purely active operational work in the queue until there is enough evidence of return value, referral readiness, or retention timing.

Record screens

Inquiry create/update and loyalty create/update screens now share the same page-based docs navigation pattern as the list views. Use the top action buttons to open this reference page without bringing back an inline side rail.

Inquiry create
Open docs before first-touch drafting
Use the form-level docs button while shaping the first operator summary, ownership, and follow-up timing so terminology stays nearby without shrinking the queue-side workflow.
Inquiry update
Keep queue reading separate from live record editing
The inquiry detail screen should stay focused on guest posture, workflow changes, and the next action. Open this docs page when an operator needs glossary or bridge guidance without losing the record header context.
Loyalty screens
Use the same route pattern across continuity work
Repeat-guest create and update screens keep the same three simple routes: help page, loyalty list, and inquiry list. This keeps orientation clear without bringing back a permanent right-side help column.

Glossary

Original request
The original guest request and identity details that this repeat-guest record should stay connected to.
Original request status
The status of the original request, used to compare whether repeat-guest follow-up started at the right time.
Repeat-guest follow-up status
The current repeat-guest status, usually a mix of overall state and stage, showing how far the follow-up work has progressed.
Repeat-booking signal
A short reading such as watch, promising, strong, or flagship that helps decide how much future follow-up effort makes sense.
Latest result
The most recent follow-up result, showing whether the guest lane moved forward, paused, was retained, or needs another review.
Latest brief status
The latest state of the follow-up brief, such as no brief, watch, pending, or prepared. It shows how ready the next step already is.
Next check-in
The next planned review time. The lists show this so operators can see timing without opening the full form.
